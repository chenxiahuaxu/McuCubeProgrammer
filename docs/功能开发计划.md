## 关键设计决策

### 1. 芯片信息面板(#3)提到最前
`connect()` 已提取完整的 `TargetInfo` 数据（flash_regions、ram、sector_size），但全部丢弃在内部日志里。做一个纯 UI 面板展示即可，零后端改动，快速出活，同时解决"连接后看不到任何反馈"的体验问题。

### 2. `read_memory()` 是基础设施阻塞点
`BackendABC` 目前没有暴露 `read_memory(address, size)` 方法。不加这个：
- 芯片内存 hex dump (#1) 做不了
- 多内存标签页 (#4) 做不了
- 变量监控 (#12-17) 全部做不了
- 选项字节 (#18-19) 的读回验证也做不了

这是一个 **unblock 整个下游** 的改动，必须在芯片查看阶段最早完成（紧随信息面板之后）。

### 3. 目标控制 + 变量监控合并为「调试」模块
两者共享相同的运行时依赖——`read_memory()` 提供数据通道，halt 机制保证读取一致性。
在目标全速执行时周期读取内存，读到的是瞬时快照，值可能不一致甚至触发总线错误。halt 模式下的读取才有意义。因此目标控制必须在变量监控之前修好，放到同一个模块下也体现这种内聚关系。

### 4. elf 解析(#5)并入调试模块
单独做 elf 解析的产物是符号表——如果不配合内存读取和监控列表，这个产物无处使用。将其作为变量监控的数据源阶段，与符号加载(#13)和周期读取(#12)协同交付。
